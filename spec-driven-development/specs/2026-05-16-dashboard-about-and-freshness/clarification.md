## Q1 -- Data freshness mechanism

**Question.** How should the live dashboard stay in sync with the repository
after a `git push` to master, given that the current Azure Container Apps
deployment is built from a snapshot image?

**Options presented:**
- (a) Document-as-expected: declare stale-until-redeploy a known limitation; no code change.
- (b) GitHub Actions OIDC auto-redeploy on push to master.
- (c) Runtime repo sync: volume-mount the repo or `git pull` on container startup.

**Decision: (b) GitHub Actions OIDC auto-redeploy on push to master.**

**Rationale (accepted from PM recommendation):**
- REC-3 workflow is already drafted in
  `spec-driven-development/specs/2026-05-16-cloud-dashboard/SECURITY-REVIEW.md`
  and `DESIGN.md` Section 6.
- Reuses the existing OIDC trust pattern -- no new attack surface.
- Bounds freshness latency to a measurable SLO (see Q3).
- Option (c) would require either a writable mount (violates immutable-image
  posture) or a startup `git pull` (adds repo credentials to the container and
  couples cold-start time to network conditions).

**Implication.** The deferred REC-3 workflow file will be committed with both
`workflow_dispatch` and `push` (master) triggers, and the federated credential
will be created per the runbook step 5 in the cloud-dashboard DESIGN.md.

## Q2 -- About-section content model

**Question.** What should drive the content of the new "About / Where we are"
section?

**Options presented:**
- (a) Fully static markdown blob hand-edited per release.
- (b) Fully dynamic, generated entirely from `exec/state.md`.
- (c) Hybrid: static purpose paragraph + dynamic PI/sprint line from `exec/state.md`.

**Decision: (c) Hybrid.**

**Rationale (accepted from PM recommendation):**
- Static paragraph carries the framework's purpose and the meta caveat (the
  dashboard tracks the framework's own progress) -- content that does not
  change per sprint.
- Dynamic PI/sprint line is pulled from `spec-driven-development/exec/state.md`
  (already auto-generated by `state_builder.py`) so the meta-claim stays
  honest with zero extra maintenance burden.

## Q3 -- Acceptance latency target

**Question.** What is the binding upper bound on the time between `git push`
to master and the change being visible on the live dashboard?

**Decision: N = 5 minutes.**

**Binding acceptance criterion (verbatim):**
> From `git push` to master, the change is visible on
> `https://state-dashboard.politehill-ac7984d9.westus2.azurecontainerapps.io/`
> within 5 minutes.

This becomes the headline acceptance criterion of `spec.md`.
