# Live UI v2 -- Specification

## 1. Problem Statement

The current v1 Bridge dashboard (`exec/state.html`, generated by `cli/state_builder.py`)
was evolved from a CLI side-output, not designed from a UI-first perspective. It serves
its purpose -- visualizing fleet state, feature lifecycle, and dispatch history -- but has
three structural gaps that this spec addresses:

1. **No navigation hierarchy.** The dashboard is a single flat page. The framework now has
   a three-tier navigation layer (HIGH_LEVEL_DEV_TRACKER -> PI INDEX -> Sprint SPEC), but
   the dashboard does not expose it. Users must leave the dashboard and navigate Markdown
   files to understand the PI/sprint structure.

2. **No sprint-first information architecture.** The v1 layout leads with Mission Progress
   (Zone A) and Fleet Roster (Zone B). The human's approved priority ranking (Q2) is:
   current sprint first, what comes next second, WIP summary third. The dashboard does not
   reflect this hierarchy.

3. **No tablet responsiveness.** The v1 layout is fixed-width, optimized for wide desktop
   screens. The human has approved desktop + tablet (768px+) as the responsive target (Q5).

Secondary gaps:

- No reserved region for future live agent visibility (the human wants a placeholder now,
  implementation later -- Q4).
- No subtle transitions; the page renders statically with no visual feedback on hover or
  load (Q6 approves fade-in and hover states).

This spec does NOT propose changing the visual identity. Per Q1, we evolve the existing
Bridge aesthetic (dark carbon, monospace, oxblood/amber/jade signals).

---

## 2. User Experience Summary

The v2 dashboard is a single-page operations console that answers the operator's primary
question -- "where are we right now?" -- within the first viewport. The current sprint
occupies the dominant visual position: its name, goal, task status, and blockers are
immediately visible. Below the sprint, a "What Comes Next" card surfaces the recommended
next action and upcoming gates. The remaining viewport depth shows WIP progress (the
feature swim lanes), PI context (a compact PI selector with sprint table), and a
placeholder region for future per-agent activity. The aesthetic is the evolved Bridge:
dark carbon background, paper-cream text, oxblood accents, monospace typography, hard
borders, no decorative chrome. At 768px the layout collapses from a two-column grid to a
single stack, preserving all content and reading order. Subtle fade-in on load and smooth
hover states make the surface feel live without creating accessibility barriers.

## 4. Layout Specification

### 4.1 Desktop (1280px+)

CSS Grid with named areas:

```css
.dashboard {
  display: grid;
  grid-template-columns: 1fr 1fr;
  grid-template-rows: auto auto auto auto auto auto auto;
  grid-template-areas:
    "topbar    topbar"
    "sprint    sprint"
    "next      wip"
    "pi        pi"
    "agents    agents"
    "feed      feed"
    "footer    footer";
  gap: 0;
  min-width: 1024px;
}
```

| Named area | Columns | Height | Content |
|-----------|---------|--------|---------|
| `topbar` | full width | 56px fixed | Top bar chrome |
| `sprint` | full width | auto (content-driven) | Current Sprint section |
| `next` | left column | auto | What Comes Next |
| `wip` | right column | auto | WIP Summary (swim lanes) |
| `pi` | full width | auto (collapsible) | PI Context navigator |
| `agents` | full width | auto (placeholder) | Agent Activity placeholder |
| `feed` | full width | max-height 480px, scrollable | Activity Feed |
| `footer` | full width | 32px fixed | Footer |

### 4.2 Tablet (768px -- 1279px)

Single-column stack. All named areas become `grid-template-columns: 1fr`.

```css
@media (max-width: 1279px) {
  .dashboard {
    grid-template-columns: 1fr;
    grid-template-areas:
      "topbar"
      "sprint"
      "next"
      "wip"
      "pi"
      "agents"
      "feed"
      "footer";
  }
}
```

Content reading order matches priority ranking -- sprint first, activity feed last.
No horizontal scrolling at 768px. Swim lane bars scale proportionally.

### 4.3 Below 768px

Not in scope (Q5 decision). The page is usable but not optimized. No layout
changes below 768px beyond natural CSS flow.

### 5.2 Current Sprint (Priority 1)

**Data shown:**
- Sprint identifier: "S{N}: {title}" (e.g., "S4: Live UI v2 Spec")
- Sprint goal: one-sentence summary from the sprint SPEC
- Task status grid: 4 counters (done, in-progress, blocked, pending) with visual bars
- Blocker list: up to 3 active blockers with short descriptions

**Data source:**
- Sprint metadata: parsed from `docs/Management/PI-{N}/INDEX.md` marker-block
  (`<!-- BEGIN auto-generated:sprints -->`) which lists sprint name, status, dispatch count
- Sprint goal: parsed from `docs/Management/PI-{N}/Sprint-{M}-{title}/SPEC.md` section
  "Sprint Goal" (first `## 1. Sprint Goal` heading)
- Task counters: computed from `dispatches` table in `ledger/fleet.db` filtered by current
  PI + sprint, grouped by outcome (NULL = in-progress, 'success' = done, stale NULL = blocked)
- Blockers: `dispatches` rows where `outcome IS NULL AND dispatched_at < (now - 4h)`,
  plus any `decisions` rows with `description LIKE '%blocked%'`

**Semantic HTML:**
```html
<section aria-labelledby="sprint-heading">
  <h2 id="sprint-heading"><!-- sprint name --></h2>
  <p class="sprint-goal"><!-- goal text --></p>
  <div role="group" aria-label="Task status">
    <dl> <!-- done/in-progress/blocked/pending with <dt>/<dd> --> </dl>
  </div>
  <div aria-label="Active blockers">
    <ul> <!-- blocker items --> </ul>
  </div>
</section>
```

**Visual treatment:**
- Background: `--bg-carbon` (flush with page)
- Sprint heading: `--ink-paper`, 22px (maps to v1 `--type-h2`), monospace
- Goal text: `--ink-paper-dim`, 14px body
- Status counters: inline `<dl>` with colored count values:
  - Done count: `--signal-jade`
  - In-progress count: `--accent-oxblood`
  - Blocked count: `--signal-amber`
  - Pending count: `--ink-paper-faint`
- Progress bar: horizontal bar under the counters, segmented by status color
- Blockers: `--signal-amber` left border (3px), `--ink-paper` text
- Padding: 24px (`--sp-6`) horizontal, 16px (`--sp-4`) vertical
- Border: 1px `--rule-line` bottom

**Responsive (tablet):** Counters stack to 2x2 grid instead of 4 inline. Progress bar
full width. Blockers remain stacked list.

#### 5.6.1 Agent Activity Data Contract (for PI-5 implementation)

When the full agent activity feature is implemented, the UI region will display:

```
Per agent:
  - agent_id: TEXT (from roster/agents.json)
  - agent_role: TEXT (principal | generic | specialist)
  - current_task: TEXT | null (from dispatches WHERE outcome IS NULL)
  - last_activity: TIMESTAMP (MAX(dispatched_at, outcome_at) from dispatches)
  - status: ENUM (working | idle | blocked | unused)
  - dispatch_count: INTEGER (COUNT from dispatches)

Aggregate:
  - agents_working: INTEGER
  - agents_idle: INTEGER
  - agents_blocked: INTEGER
```

The status derivation rules (from v1 DESIGN.md):
- `working`: has a dispatch with `outcome IS NULL` and `dispatched_at > now - 4h`
- `idle`: last dispatch completed, or no dispatches in 1h
- `blocked`: has a dispatch with `outcome IS NULL` and `dispatched_at < now - 4h`
- `unused`: no dispatches in ledger for this agent

**Data source (PI-5):** New query joining `roster/agents.json` with `dispatches` table.
May require a new `agents` table in fleet.db (see Section 10, Data Requirements).

## 6. Navigation Integration (Q3)

The dashboard integrates the three-tier Management navigation layer as follows:

### Tier mapping

| Tier | Source | Dashboard surface | Interaction |
|------|--------|-------------------|-------------|
| 1 (Tracker) | `docs/HIGH_LEVEL_DEV_TRACKER.md` | PI tab selector in Section 5.5 | Click PI tab to switch context |
| 2 (PI Index) | `docs/Management/PI-{N}/INDEX.md` | Sprint table in Section 5.5 | View sprints, status, dispatch counts |
| 3 (Sprint Detail) | `docs/Management/PI-{N}/Sprint-{M}/SPEC.md` | External link from sprint table row | Opens Markdown in repo viewer |

### Parsing requirements for state_builder.py

1. **PI list:** Already parsed by `load_pis()` from `constitution/roadmap.md`. No change needed.

2. **Sprint table per PI:** New parsing function needed. Must read the marker-block
   `<!-- BEGIN auto-generated:sprints -->...<!-- END auto-generated:sprints -->` from
   `docs/Management/PI-{N}/INDEX.md` and extract the table rows (Sprint, Title, Status,
   Dispatches, Last Outcome, Detail link).

3. **Current sprint detection:** The current sprint is the first sprint in the current PI
   whose status is NOT "DONE" and NOT "Proposed". Falls back to the first sprint if all
   are DONE or Proposed.

## 8. Accessibility

### 8.1 Contrast targets

All text-on-background combinations meet WCAG 2.1 AA minimum (4.5:1 for normal text,
3:1 for large text and UI components):

| Foreground | Background | Ratio | Passes |
|-----------|-----------|-------|--------|
| `--ink-paper` (#e8e4d8) | `--bg-carbon` (#0a0a0a) | 15.8:1 | AAA |
| `--ink-paper-dim` (#b8b4a8) | `--bg-carbon` (#0a0a0a) | 10.6:1 | AAA |
| `--ink-paper-faint` (#8a8678) | `--bg-carbon` (#0a0a0a) | 5.8:1 | AA |
| `--ink-paper` (#e8e4d8) | `--bg-graphite` (#141413) | 13.9:1 | AAA |
| `--ink-paper-dim` (#b8b4a8) | `--bg-graphite` (#141413) | 9.3:1 | AAA |
| `--accent-oxblood` (#ce2029) | `--bg-carbon` (#0a0a0a) | 5.1:1 | AA (large + UI) |
| `--signal-amber` (#d29a3b) | `--bg-carbon` (#0a0a0a) | 8.9:1 | AAA |
| `--signal-jade` (#6fa37a) | `--bg-carbon` (#0a0a0a) | 7.1:1 | AAA |
| `--ink-paper` (#e8e4d8) | `--accent-oxblood` (#ce2029) | 3.1:1 | AA (large + UI) |

Note: `--accent-oxblood` on `--bg-carbon` at 5.1:1 passes AA for large text (18px+)
and UI components but NOT for normal body text. Oxblood is used only for accents (borders,
headings, active indicators), never for body paragraphs.

Each section uses a `<section>` with `aria-labelledby` pointing to its `<h2>`. The page

```
  <h2> What Comes Next
  <h2> Work in Progress
  <h2> Program Increments

```

### 8.3 Landmark regions

| HTML element | ARIA role | Content |
|-------------|----------|---------|
| `<header>` | `banner` | Top bar with title, PI pills, freshness |
| `<main>` | `main` | Sections 1-6 (all content between header and footer) |
| `<nav>` | `navigation` | PI selector (within top bar and PI Context section) |

### 8.4 Keyboard navigation flow

Tab order follows the visual priority ranking (top to bottom, left to right):

1. Skip-to-main link (hidden, appears on focus)
2. Top bar: PI pills (left/right arrow within group)
3. Current Sprint section (focusable heading)
4. What Comes Next: action card link
5. WIP Summary: feature table (focusable rows)
6. PI Context: PI tabs (left/right arrow), sprint table rows
7. Agent Activity: fleet stat values
8. Activity Feed: scrollable region (tab to enter, arrow keys to scroll)
9. Footer

### 8.5 Focus indicators

All interactive elements use a visible focus ring:
- `outline: 2px solid var(--ink-paper)` (15.8:1 contrast on carbon)
- `outline-offset: 2px`
- No `outline: none` anywhere in the stylesheet

### 8.6 Screen reader support

- All data visualizations (progress bars, swim lanes) have `aria-label` with
  text equivalent (e.g., "fleet-cli: IMPLEMENT stage, 4 of 9 stages complete")
- The activity feed uses `role="log"` with `aria-live="off"` (static page, not
  live-updating in v2)
- PI tabs use `role="tablist"` / `role="tab"` / `role="tabpanel"` pattern
- Status colors are never the sole indicator; text labels accompany all colored elements

### 8.7 Motion and animation

Per Q6 (subtle transitions):
- Page load: `opacity: 0 -> 1` fade-in, 300ms ease, on `<main>`
- Hover states: `background-color` transition 150ms ease on table rows and feed items
- All transitions respect `prefers-reduced-motion: reduce` -- when set, all transitions
  are instant (`transition-duration: 0s`)
- No auto-playing animations, no infinite loops, no content that flashes

## 9. Tech Constraints

### 9.1 Stdlib-only compliance

This spec is designed for implementation using Python standard library and plain HTML/CSS
only, per `constitution/tech-stack.md`:

- **HTML generation:** `state_builder.py` using string concatenation or `html.escape()`
  (existing pattern)
- **CSS:** Embedded `<style>` block in the generated HTML (existing pattern)
- **HTTP serving:** `http.server.ThreadingHTTPServer` (existing pattern)
- **Data access:** `sqlite3` for ledger, `pathlib` for filesystem, `json` for roster
- **No JavaScript frameworks:** No React, Vue, Angular, HTMX, Alpine
- **No CSS preprocessors:** No Sass, Less, PostCSS
- **No build tools:** No webpack, vite, esbuild
- **No web fonts loaded from CDN:** System monospace stack only
  (`ui-monospace, 'Cascadia Code', 'JetBrains Mono', 'Fira Code', monospace`)

### 9.2 Font stack

The v1 DESIGN.md recommended Berkeley Mono and PP Supply Mono (external fonts requiring
self-hosting). Under the stdlib constraint, v2 uses a system monospace stack:

```css
font-family: ui-monospace, 'Cascadia Code', 'JetBrains Mono', 'Fira Code',
             'Source Code Pro', monospace;
```

This avoids any WOFF2 file hosting or font-loading CSS. The visual identity (monospace,
operations console) is preserved through system fonts already installed on developer
machines.

### 9.3 No new Python dependencies

No additions to the approved dependency list in `tech-stack.md`. The implementation
uses only: `sqlite3`, `json`, `pathlib`, `argparse`, `html`, `re`, `datetime`,
`http.server`, `dataclasses`, `subprocess`, `socket`, `webbrowser`, `sys`.

fleet.db. That schema change will be spec'd in PI-5, not here.

## Appendix B: Open Questions (for Architect review)

1. **Sprint table parsing robustness.** The marker-block format in INDEX.md is
   auto-generated by `state_builder.py build-index`. If the format changes, the
   HTML parser breaks. Should we define a schema contract for the marker-block,
   or parse the filesystem directly instead?

2. **Collapsible PI sections.** The `<details>`/`<summary>` HTML elements provide
   native collapsibility without JavaScript. However, their styling is limited
   and browser-inconsistent. Is this acceptable, or should we use CSS-only
   toggle patterns (checkbox hack)?

3. **Activity Feed scroll.** The v1 feed renders all events. With a 480px max-height
   and scroll, we need to decide: render all events in HTML (simple, potentially
   large DOM), or cap at N events (e.g., 50) with a "show more" that requires
   page regeneration?
