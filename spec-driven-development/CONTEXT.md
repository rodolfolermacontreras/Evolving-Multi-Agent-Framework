# CONTEXT -- Shared Language for Day-to-Day Agent

This file captures the shared vocabulary, assumptions, and conventions that all agents
must understand before working on this project. It follows the Matt Pocock pattern: a
single document that grounds all participants in the same mental model.

Updated by `grill-with-docs` sessions and ADRs. Acts as a living glossary.

---

## Project Identity

- Day-to-Day Agent is a personal AI-powered work management system for a single user (Rodolfo)
- It integrates Microsoft 365 data (calendar, tasks, emails) with local task and project management
- The app runs locally on Windows, deployed via Docker for portability
- There is no multi-tenant, no authentication wall, no external user base

## SDD Framework Terms

| Term | Definition |
|------|------------|
| **PI** | Program Increment -- a 10-week planning horizon containing 5 sprints (symbolic; AI fleet compresses wall-clock time) |
| **Sprint** | Two-week development cycle within a PI (symbolic cadence) |
| **Spec** | A formal feature specification document in `specs/YYYY-MM-DD-feature-name/spec.md` |
| **Plan** | Implementation plan co-authored by Architect and SW Dev, lives next to the spec |
| **Task** | An atomic unit of work (1-3 files max) derived from a plan, tagged with [P]/[S]/[AFK]/[HITL] |
| **Gate** | A formal checkpoint between lifecycle phases requiring approval before proceeding |
| **AFK** | Away From Keyboard -- a task safe for fully autonomous agent execution |
| **HITL** | Human In The Loop -- a task that requires a human decision before completion |
| **Fleet** | The pool of generic worker agents available for parallel dispatch |
| **Dispatch** | A single assignment of one task to one worker agent, recorded in the ledger |
| **Worker** | A generic agent (Developer, UX Designer, Data Scientist, QA Engineer) |
| **Principal** | A senior agent role with decision authority (Executive Manager, Product Manager, Architect, Software Developer) |

## Project-Specific Terms

| Term | Definition |
|------|------------|
| **Engine** | The central orchestrator in `agent/engine.py`. Lazy singleton. Owns the file watcher, scheduler, git auto-commit, and WebSocket broadcast queue. |
| **World State** | The aggregated context dict assembled by `agent/world_state.py` and passed to all LLM prompts. Single source of truth for what the agent knows. |
| **Board** | The project board view, driven by `MAIN_PROJECTS` in `agent/board.py`. Controls which projects appear in status reports and the board page. |
| **Accountability** | The daily work log system. Quick entries captured via `/accountability` route, stored in `.accountability_log.json` (migrating to SQLite). |
| **Inbox** | The `inbox/` directory. Files dropped here are picked up by the FileWatcher, deduplicated via SHA256, and routed to the appropriate workflow. |
| **Watcher** | The `FileWatcher` component inside the Engine. Monitors `inbox/` for new files and triggers workflow dispatch. |
| **Dotfiles** | Legacy JSON stores in `agent/` (e.g., `.accountability_log.json`, `.ideas.json`). Protected by `file_lock()`. Migration to SQLite is ongoing (D1). |
| **Patched Settings** | The `patched_settings` pytest fixture in `conftest.py`. Monkeypatches `settings.paths` to `tmp_path` so tests never touch real data. |
| **Safe Path** | The `safe_path(base, *parts)` function in `agent/routes/__init__.py`. Prevents path traversal attacks on user-supplied file names. |
| **World State** | See above. Do not query data sources directly in prompt builders. Always call `get_world_state()`. |
| **Sidecar** | A JSON override file in `export/overrides/` that manually overrides fields in a generated status report without regenerating it. |
| **Status Report** | The bi-weekly HTML report generated by `agent/status_report.py` + `agent/status_report_renderer.py`. Output: `export/weekly_status_YYYY-MM-DD.html`. |
| **Ledger** | `spec-driven-development/ledger/fleet.db` -- the SQLite source of truth for all fleet dispatches, agent records, and skill assignments. |
| **Two-Folder Split** | The architectural rule that `.github/` holds Copilot-native files (agents, skills, prompts, instructions) and `spec-driven-development/` holds everything else. |

## Lifecycle Shorthand

```
IDEA -> BACKLOG -> CLARIFY -> SPEC -> PLAN -> TASKS -> IMPLEMENT -> REVIEW -> DONE
```

- `/triage` -- groom an idea, RICE score, assign priority
- `/clarify` -- structured clarification session (one question at a time, with recommendation)
- `/spec` -- generate a feature spec from clarified requirements
- `/plan` -- generate an implementation plan from an approved spec
- `/tasks` -- decompose a plan into tagged, batched tasks
- `/fleet` -- parallel dispatch of [P][AFK] tasks to workers
- `/implement` -- execute a single task (TDD: test first, then implement)
- `/qa` -- validate implementation against spec
- `/retro` -- sprint retrospective
