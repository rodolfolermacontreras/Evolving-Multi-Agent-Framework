## Approach Summary

Implement SDD-022 as a new stdlib-only CLI, `spec-driven-development/cli/taskstoissues.py`, wrapped by a new `/taskstoissues` prompt. The CLI reads a spec directory's `tasks.md`, renders deterministic issue payloads, and mirrors tasks into GitHub Issues on explicit invocation. Dry-run is default; write mode requires `--apply` plus env-var token auth.

`tasks.md` remains authoritative. The bridge uses per-spec-dir `issue-map.json` for identity mapping and writes non-mutating conflict reports when remote tracker state disagrees with local task state. The v1 provider boundary proves GitHub end to end and preserves an ADO-compatible contract/test double without making live ADO execution a Sprint 8 close criterion.

---

## Design Decisions

1. **New standalone CLI**: `cli/taskstoissues.py` rather than extending `fleet.py`, `state_builder.py`, or ledger tooling. This keeps issue-sync behavior isolated and easy to test.
2. **Provider boundary, GitHub-first**: Define provider operations around create/update/get issue state. Implement GitHub live provider first. Add ADO-compatible provider contract or test double to avoid redesign later.
3. **Dry-run default**: All commands render intended writes without side effects unless `--apply` is present.
4. **Mapping file in spec dir**: `issue-map.json` lives beside `tasks.md` to keep identity state portable and reviewable without a ledger migration.
5. **Generated body section**: Issue bodies include a generated marker/region so human comments and non-generated tracker discussion are not clobbered.
6. **Conflict report over mutation**: Remote drift writes `issue-conflicts.md` and exits non-zero. It never rewrites `tasks.md`.
7. **Strict stdlib HTTP**: Use `urllib.request`, `urllib.parse`, `urllib.error`, and `json`; add import tests to guard Article V.

## Implementation Phases

### Phase 1: Local Task Contract

Build and test the parser for `tasks.md`, including task headings, status, story/requirement traceability, file scope, blocked files, acceptance criteria, and verification blocks. This phase must treat parse failures as expected CLI errors and report them clearly.

### Phase 2: Deterministic Rendering

Render GitHub-compatible issue title, generated body, labels, status, and source links from parsed task records. The renderer must be pure and deterministic so tests can compare exact output.

### Phase 3: Mapping and Idempotency

Implement `issue-map.json` load/write behavior with stable JSON formatting. Use the mapping to decide create vs update and to prevent duplicate issue creation on repeated runs.

### Phase 4: Providers and Auth

Implement GitHub provider request construction and mocked transport tests using stdlib `urllib.*`. Add provider protocol boundaries and ADO-compatible test double. Add env-var auth resolution and redaction tests.

### Phase 5: Conflict Detection

Compare local task status against remote status from mapped issues. When conflict exists, write `issue-conflicts.md`, exit with conflict code, and do not mutate `tasks.md`.

### Phase 6: CLI + Prompt Wrapper

Expose subcommands and flags through argparse. Add `.github/prompts/taskstoissues.prompt.md` with dry-run-first operating instructions and explicit lifecycle placement after `/tasks` lock.

### Phase 7: Validation Close-Out

Run targeted tests, import scan, `schema_lint`, and full pytest. Check every REQUIRED item in `validation.md` before marking SDD-022 DONE.

## Parallelization Guidance

The core CLI file is a shared surface, so the safest implementation is mostly sequential. Limited parallel work is possible after the parser/renderer interfaces are stable:

- Prompt wrapper can be drafted in parallel with provider implementation if it only references the CLI contract in this plan.
- Import-scan tests can be drafted in parallel with provider implementation.
- ADO provider test double can be drafted after the provider protocol is defined, without touching GitHub request code.

Do not split multiple workers across `cli/taskstoissues.py` at the same time unless the Software Developer creates explicit non-overlapping edit regions or a handoff protocol.

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| GitHub REST behavior differs from mocked transport | Medium | Medium | Use request-construction tests plus optional live smoke with owner-provided token |
| Token leakage in logs or mapping files | Low | High | Redaction helper + tests; never store token-bearing values |
| Parser too brittle against task-list variants | Medium | Medium | Test against current SDD-022 tasks and at least one completed prior tasks.md fixture |
| Accidental bidirectional mutation pressure | Medium | High | Conflict report only; tests assert `tasks.md` unchanged |
| ADO fast-follow boundary too vague | Medium | Medium | Define provider protocol and ADO dry-run/test fixture now |
| Third-party dependency pressure | Low | High | Import scan and validation item V-12 |
