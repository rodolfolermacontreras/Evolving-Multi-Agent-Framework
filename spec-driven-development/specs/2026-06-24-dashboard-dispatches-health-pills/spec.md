## Problem Statement

The framework dashboard (`exec/state.html`, generated by `cli/state_builder.py`) renders the sprint table, lifecycle zone, decisions, and ledger summary, but it does **not** show:

1. **What the fleet actually did.** The `dispatches` rows in `ledger/fleet.db` (agent, role, task, outcome, timestamps, grouped by feature/sprint) are read for summary counts but never displayed as a browsable card. A human cannot see the dispatch history on the dashboard.
2. **Whether the framework's own invariants are healthy.** There is no at-a-glance signal for constitution-version consistency, skill-frontmatter validity, fleet-ledger reachability, or tracker freshness. Drift in any of these is invisible until something breaks downstream.

SDD-037 adds two additive, read-only dashboard surfaces -- a **Dispatches card** and a **health-pills strip** -- so the operator sees both fleet activity and framework health in one pane, with click-through to failure detail. Both surfaces degrade gracefully and never block dashboard generation.

## Goal

Render, on every `state_builder` build tick, (a) a Dispatches card grouped by feature then sprint from the existing single ledger read, and (b) a header strip of four green/yellow/red health pills (constitution semver, skill validity, ledger reachability, stale-tracker) with server-rendered click-through detail -- all additive `inject_*` post-processors that leave the Article X locked S1 functions byte-identical and introduce no JavaScript, no new schema, no new ledger API, and no new gate.

## Non-Goals

- NG-1: No new ledger read API in `ledger/__init__.py` (it stays empty); no new ledger table or column.
- NG-2: No second `sqlite3.connect`; the ledger is read exactly once per `build()` tick (Q-B).
- NG-3: No health check becomes a gate, CI failure, or non-zero exit (Q-F); pills are indicators only.
- NG-4: No JavaScript, no client-side routing, no new server route (Q-D, Q-E); `script tags:0` is preserved.
- NG-5: No modification to the five Article X locked S1 functions (`render_html`, `load_sprint_table`, `load_sprint_goal`, `detect_current_sprint`, `load_decisions`); `TestS1FootprintLockGuard` must keep passing.
- NG-6: No write to `constitution/**` (Article VIII); the semver pill reads only.
- NG-7: No new third-party dependency (Article V stdlib-only).
- NG-8: No change to the `dispatches`/`decisions` data written by `cli/fleet.py`; this feature only reads.
- NG-9: No sprint-close smoke run, deployment-workflow change, or push (those belong to F-29).

## Requirements (RFC-2119)

### Dispatches card

- FR-1: The system MUST render a Dispatches card on `state.html` showing, per dispatch row: agent (`agent_id`), role (`agent_role`), task (`task_id` + `task_title`), status (`outcome`, or a "pending" label when null), and when (`outcome_at` if present else `dispatched_at`).
- FR-2: The card MUST group rows by `feature_dir` then by `sprint`.
- FR-3: The card MUST source its rows from the **existing** `load_ledger()` read (the single per-tick `LedgerView`); it MUST NOT open a second database connection and MUST NOT add a public ledger package API. F-28 MAY additively widen the `LedgerView` returned by that single call (e.g. an additional grouped/all-rows field populated inside the same connection) without changing `recent`.
- FR-4: When the ledger is reachable but has no rows, the card MUST render an empty state with a plain-language note (e.g. "No dispatches recorded yet.").
- FR-5: When the ledger is unreachable (`LedgerView.available is False` -- missing `fleet.db` or `sqlite3.Error`), the card MUST render a disabled state with a plain-language reason and MUST NOT raise.

### Health-pills strip

- FR-6: The system MUST render a header strip of exactly four health pills, each in one of three states: green, yellow, red.
- FR-7: **Constitution-semver pill** MUST read each `constitution/*.md` frontmatter `version:` (read-only) and report green when every file has a present, quoted, valid-semver `X.Y.Z`; yellow when a version is present and parseable but unquoted; red when any file is missing a version or carries an unparseable version.
- FR-8: **Skill-validity pill** MUST reuse `schema_lint.check_skill` over `.github/skills/**/SKILL.md`, reporting green on zero findings and red on any finding. It MUST NOT reimplement skill validation.
- FR-9: **Ledger-reachability pill** MUST report green when `LedgerView.available` is true and red otherwise, surfacing the reason; it MUST consume the same single per-tick `LedgerView` as the Dispatches card.
- FR-10: **Stale-tracker pill** MUST compute the age in whole days of the latest ISO-8601 date token in `exec/sprint-progress.md` relative to the build date, reporting green for age <= 7, yellow for 8-14, red for > 14, and yellow ("freshness unknown") when no date parses. The threshold N MUST be 7 days.
- FR-11: Each non-green pill MUST be an anchor to a server-rendered in-page detail section (`#health-detail-<check>`) that lists the concrete failures; green pills MUST render no link and need no detail section.
- FR-12: The pill strip MUST contain no JavaScript and MUST NOT increase the document's `<script>` tag count (which MUST remain 0).

### Cross-cutting

- FR-13: Both surfaces MUST be implemented as additive `inject_*` post-processors wired in `build()` after `inject_lifecycle_html`; they MUST NOT modify any of the five Article X locked S1 functions.
- FR-14: All health checks MUST be read-only indicators: they MUST NOT raise out of the build, MUST NOT alter `build()`'s exit behavior, and MUST NOT be wired into any lint/pytest/CI gate (Q-F). Internal exceptions MUST degrade to red (or yellow for the stale-unknown case) with a reason.
- FR-15: The implementation MUST be stdlib-only (Article V).
- FR-16: After implementation (F-28), `python spec-driven-development/cli/schema_lint.py` MUST exit 0 and the full `cli/` pytest suite MUST be >= the pre-F-28 passed count plus the new tests (the two known skips unchanged).

## Acceptance Criteria

- AC-1 (FR-1, FR-2): Given a populated `fleet.db`, the rendered `state.html` contains a Dispatches card with rows grouped by feature then sprint, each row showing agent, role, task id+title, status, and a timestamp. (UI-Variant render; Strict grouping correctness.)
- AC-2 (FR-4): Given a reachable ledger with zero rows, the card renders an empty-state note and the build succeeds. (Strict + UI-Variant.)
- AC-3 (FR-5): Given a missing or unreadable `fleet.db`, the card renders a disabled state with a reason and `build()` does not raise. (Strict.)
- AC-4 (FR-3, Q-B): Across one `build()` tick, exactly one `sqlite3.connect` to `fleet.db` occurs (the existing `load_ledger`); the Dispatches card and ledger pill both consume that single `LedgerView`. (Strict.)
- AC-5 (FR-6, FR-12): The rendered header contains exactly four pills; the document's `<script>` count remains 0. (UI-Variant render + Strict no-JS invariant.)
- AC-6 (FR-7): The constitution-semver pill is green for the current quoted-semver constitution, yellow when a version is unquoted, and red when a version is missing/unparseable -- and never writes any constitution file. (Strict.)
- AC-7 (FR-8): The skill-validity pill is green when `check_skill` yields zero findings and red when at least one skill produces a finding, via reuse of `check_skill` (no reimplementation). (Strict.)
- AC-8 (FR-9): The ledger-reachability pill is green when `LedgerView.available` is true and red (with reason) otherwise. (Strict.)
- AC-9 (FR-10): The stale-tracker pill returns green / yellow / red for tracker ages of <=7 / 8-14 / >14 days respectively, and yellow when no date parses, with N fixed at 7. (Strict.)
- AC-10 (FR-11, FR-12, Q-D): Each non-green pill links to a server-rendered `#health-detail-<check>` section listing concrete failures; green pills have no link; no `<script>` is added. (UI-Variant render + Strict no-JS.)
- AC-11 (FR-14, Q-F): Forcing each check to fail internally degrades its pill to red/yellow with a reason and does NOT raise, does NOT change `build()`'s exit, and is NOT wired into any gate. (Strict.)
- AC-12 (FR-13, NG-5): `TestS1FootprintLockGuard` passes unchanged (the five locked functions' source SHA-256 match the 257b081 goldens); both new surfaces are additive injectors wired after `inject_lifecycle_html`. (Strict.)
- AC-13 (FR-15, FR-16): The implementation imports only the standard library; `schema_lint` exits 0; the full `cli/` suite passes at >= baseline + new tests. (Strict.)

## Data Model Changes

- None to any persisted store. No new ledger table or column; the `dispatches` schema is read as-is. The only in-memory shape change is an **additive field on the `LedgerView` dataclass** (e.g. a grouped/all-rows accessor) populated within `load_ledger`'s existing single connection; `recent`, `recent_success`, `blockers`, and `available` are unchanged so `build()`'s existing `"dispatches": len(ledger.recent)` report is preserved.

## API Changes

- No public API change. `ledger/__init__.py` remains empty (no new ledger package API). All new functions are module-internal helpers in `cli/state_builder.py` (`inject_dispatches_html`, `inject_health_pills_html`, and the four health-check helpers), consistent with the existing `inject_*` family.

## Test Strategy

- Unit tests (Strict) for: the additive ledger widening (grouping correctness; single-connection; available/unavailable), each of the four health-check helpers (green/yellow/red boundaries + degradation), the indicators-not-gates invariant (no raise, no exit change), and the no-JS / S1-lock invariants.
- Render tests (UI-Variant, delta-eligible) for: Dispatches card markup grouping, empty/disabled card states, the four-pill strip, pill colors, and click-through detail-section presence/anchors.
- Reuse `tmp_path`-style isolated fixtures for synthetic `fleet.db`, synthetic constitution dirs, synthetic skills, and synthetic `sprint-progress.md` with controlled dates. No real LLM, no network.
- `TestS1FootprintLockGuard` is run unchanged and MUST pass (AC-12).

## Out of Scope

- Promoting any health pill to a blocking gate (future, separately governed).
- A queryable public ledger API or ledger schema/migration changes (NG-1).

## Risks

- R-A: **Accidental S1-lock breakage.** Editing inside a locked function while wiring injectors. Mitigation: injectors and helpers are new top-level functions; only `build()` (non-locked) gains wiring lines; `TestS1FootprintLockGuard` is the guard (AC-12).
- R-B: **A second ledger read sneaks in.** A helper opening its own `sqlite3.connect`. Mitigation: FR-3 + AC-4 assert a single connection per tick; widening happens inside `load_ledger`'s existing connection only.
- R-C: **A health check raises and aborts the build.** Mitigation: FR-14 + AC-11 require each check be wrapped to degrade gracefully; tested by forcing failures.
- R-D: **Constitution accidentally written.** Mitigation: the semver pill uses read-only `parse_frontmatter`; FR-7/AC-6 and Article VIII forbid writes; tests assert no constitution mutation.
- R-E: **JavaScript creeps in via click-through.** Mitigation: FR-11/FR-12 mandate server-rendered anchors; AC-5/AC-10 assert `<script>` count stays 0.
